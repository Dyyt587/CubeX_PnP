# UI 卡顿问题修复复盘

日期：2026-03-13

## 1. 问题现象
- 页面在运行过程中会周期性出现短暂卡顿。
- 用户反馈即使不打开摄像头，卡顿仍然存在。
- 初期观测到卡顿存在固定时间间隔特征。

## 2. 影响范围
- 主界面交互流畅度下降。
- 鼠标拖拽、滚动、点击反馈会在卡顿瞬间变慢。
- 影响用户对系统稳定性的主观感受。

## 3. 排查过程
### 3.1 第一阶段：排查摄像头链路
- 检查 CameraDeviceManager 与 CameraScanManager 的定时逻辑。
- 发现 CameraScanManager 默认启动后每 2 秒执行一次 PowerShell 与 WMI 扫描。
- 发现 CameraDeviceManager 还存在 3 秒轮询扫描。

### 3.2 第二阶段：确认问题与摄像头开启状态关系
- 用户明确反馈不开摄像头时也会卡顿。
- 说明卡顿源不止图像处理链路。

### 3.3 第三阶段：定位非摄像头周期任务
- 检查 SerialPortManager。
- 发现 SerialPortManager 每 2 秒在主线程同步执行 PowerShell，并调用 waitForFinished。
- 该行为会直接阻塞主线程，符合周期性卡顿特征。

## 4. 根因总结
根因是多个周期性系统扫描任务触发了高开销操作，尤其是 PowerShell 同步调用导致主线程阻塞。

重点根因：
- SerialPortManager 在 UI 线程同步扫描串口。

次要风险点：
- CameraDeviceManager 轮询扫描。
- CameraScanManager 周期性扫描策略过于激进。

## 5. 修复方案与落地
### 5.1 串口扫描线程化（关键修复）
修改文件：
- src/helper/SerialPortManager.h
- src/helper/SerialPortManager.cpp

改动内容：
- 新增后台工作线程与扫描工作对象。
- 定时器触发时仅发出扫描请求信号。
- PowerShell 扫描在工作线程执行。
- 主线程仅接收扫描结果并更新端口列表。

效果：
- 消除主线程被 waitForFinished 周期性阻塞的问题。

### 5.2 摄像头扫描轮询收敛
修改文件：
- src/helper/CameraDeviceManager.cpp

改动内容：
- 去掉 3 秒轮询扫描。
- 保留设备变更事件驱动与按需触发扫描。

效果：
- 减少不必要周期任务，降低系统抖动概率。

## 6. 验证结果
- 用户反馈问题已解决。
- 在不开摄像头场景下，周期性卡顿消失。

## 7. 经验与改进建议
1. 避免在 UI 线程执行外部进程并同步等待结果。
2. 周期扫描优先改为事件驱动，必要时再做低频兜底扫描。
3. 对所有定时任务标注用途、频率与线程归属，建立统一性能审查清单。
4. 对可能触发系统调用的逻辑增加耗时日志，便于快速定位下一次卡顿问题。

## 8. 后续可选优化
1. 给 SerialPortManager 增加防重入保护，避免扫描任务堆积。
2. 为扫描链路增加耗时统计与节流策略。
3. 对主页面定时器做统一梳理，建立性能基线数据。
